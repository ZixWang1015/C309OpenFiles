# 这里我记录平时使用Codex的一些心得和经验

## 总结
- 经过一段时间的使用，我最大的心得是Codex非常厉害，但是也需要用心的调整与他沟通的方式。
- 对于强度一般的任务，可以直接考虑GPT5.4中档 + 原始运行速度
- 对于高强度的任务，可以让Codex给出最理想的解决方案，然后用DeepSeek-V4 Pro加上Copilot的Agent框架来尝试编码和运行
- 对于最简单的Git操作和督工代码运行环节，可以考虑DeepSeek-V4 Flash，我觉得已经够用了

## 交互技巧
1. 当上下文快满了，我们应该引导Codex快速总结当下的任务和经验，以便在新窗口中无缝衔接，可以参考的Prompt为：
   ```
   当前上下文快满了。请不要继续写新功能，我们先停下来，请总结当前任务状态在当前路径下构建文件./docs/codex-handoff.md，内容包括： 

   1. 当前任务目标
   2. 已经完成的修改
   3. 仍未完成的问题
   4. 修改过的关键文件及每个文件的作用
   5. 当前 git status / git diff 的摘要
   6. 已运行过的测试命令及结果
   7. 当前失败的测试、报错、阻塞点
   8. 下一步最应该做什么
   9. 不要重复尝试的错误方向
   10. 继续工作时应该优先阅读的文件
    
   请注意，这里的记录应该以任务简报的形式突出重点，不要写的太过于冗长。写完后即可停止，并且将这个.md文件也加入到.gitignore内部。
   ```

   然后在打开新窗口时，快速引导Codex马上聚焦到当前的任务中：
   ```
   我们继续当前 repo 的任务。请先不要改代码。

   请先阅读：./docs/codex-handoff.md
    
   然后用 5-10 行总结你理解的当前状态，列出下一步计划，等我确认后再修改代码。
   ```

2. 利用Codex复现代码时，最后手动提醒它注意一切可以注意的细节，不然很容易忽略具体的实验细节，最后的结果就是仅仅代码跑通而已，注意进行谨慎的Prompt：
   ```
   请注意，开始代码运行之前，请仔细查看原文https://arxiv.org/pdf/2601.18734中记录的具体实验设置和细节
   ```

3. 有时智力较低或者被降智的模型，如DeepSeek-V4-Flash经常会未经允许调整你的虚拟环境，然后覆盖掉某些关键的依赖包等，造成破坏。针对这一点，你可以尝试写一个/create-instructions:
   ```
   在本工作区执行任何任务时，禁止在未得到我明确确认前修改虚拟环境或项目依赖的版本与安装状态。包括但不限于：升级、降级、删除、重装、覆盖安装依赖库，运行 pip install/upgrade/uninstall、conda install/update/remove、poetry                   update/add/remove、uv add/remove/sync、修改 requirements.txt、pyproject.toml、poetry.lock、uv.lock、environment.yml 等会改变依赖版本或环境状态的操作。

   如果你判断某个任务可能需要新增、删除或更新依赖，请先跳过该依赖变更，尽量继续完成其余不依赖环境修改的部分。任务结束时单独列出：
   1. 建议变更的库名和当前/目标版本；
   2. 为什么需要变更；
   3. 计划执行的具体命令或文件修改；
   4. 不执行该变更可能带来的影响。
   只有在我明确回复确认后，才可以执行这些依赖相关操作。
   ```

4. 需要注意Codex对于显存的使用，它经常会启动多个测试脚本来进行同步测试，很有可能将现有的进程压缩或Kill掉，因此我们应该在Prompt中向Codex传达清楚：
   ```
   测试时尽量少用多线程并发测试，可以考虑开单进程按顺序去测试，并且batch size不要设置的特别高，整体考虑压缩显存，最多不要超过CUDA Device0和1
   ```
